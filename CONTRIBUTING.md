# Contributing to AppsMobs

Thank you for your interest in contributing to AppsMobs! We welcome contributions from the community. This document provides guidelines and instructions for contributing.

## Code of Conduct

By participating in this project, you agree to maintain a respectful and inclusive environment:

- **Be Respectful**: Treat all contributors with respect and kindness
- **Be Inclusive**: Welcome newcomers and help them learn
- **Be Constructive**: Provide constructive feedback and suggestions
- **Be Professional**: Maintain professional communication at all times

## How to Contribute

### Reporting Bugs

We appreciate detailed bug reports that help us reproduce and fix issues quickly.

**Before Reporting**:
1. Check if the bug already exists in [Issues](https://github.com/AppsMobs/AppsMobs/issues)
2. Search closed issues to see if it was already fixed
3. Ensure you're using the latest version of AppsMobs

**When Reporting**:
1. Use the [Bug Report template](https://github.com/AppsMobs/AppsMobs/issues/new?template=bug_report.md)
2. Provide a clear, descriptive title
3. Include all relevant information:
   - Clear description of the bug
   - Steps to reproduce (numbered list)
   - Expected behavior
   - Actual behavior
   - Environment details (AppsMobs version, Windows version, device model, Android version)
   - Screenshots or error messages (if applicable)
   - Any relevant logs or error traces

### Suggesting Features

We value feature suggestions that improve AppsMobs for all users.

**When Suggesting**:
1. Use the [Feature Request template](https://github.com/AppsMobs/AppsMobs/issues/new?template=feature_request.md)
2. Clearly describe the feature and its use case
3. Explain why it would be useful
4. Provide examples or mockups if possible
5. Consider alternatives and trade-offs

### Contributing Code

While AppsMobs is proprietary software, we welcome contributions to:
- Documentation improvements
- Example scripts
- Bug fixes for public components
- Translation/localization

**Process**:
1. **Fork** the repository
2. **Create a branch** (`git checkout -b feature/amazing-feature`)
3. **Make changes** following our coding standards
4. **Test** your changes thoroughly
5. **Commit** with clear, descriptive messages
6. **Push** to your fork (`git push origin feature/amazing-feature`)
7. **Open a Pull Request** with a detailed description

### Code Style Guidelines

- **Python**: Follow [PEP 8](https://www.python.org/dev/peps/pep-0008/) style guide
- **Comments**: Add clear comments for complex logic
- **Documentation**: Update relevant documentation with your changes
- **Commit Messages**: Write clear, descriptive commit messages
- **Testing**: Test your changes before submitting

### Development Guidelines

- **Platform**: Test on Windows 10/11 (64-bit)
- **Compatibility**: Ensure backward compatibility when possible
- **Documentation**: Update documentation for any user-facing changes
- **Examples**: Add or update example scripts if relevant

### Pull Request Process

1. **Description**: Provide a clear description of your changes
2. **Testing**: Describe how you tested your changes
3. **Breaking Changes**: Clearly mark any breaking changes
4. **Documentation**: Ensure documentation is updated
5. **Review**: Respond to feedback and requested changes promptly

## Questions?

- **General Questions**: Open an issue with the "question" label
- **Support**: Contact support@appsmobs.com
- **Security Issues**: See [SECURITY.md](SECURITY.md) for security reporting

## Recognition

Contributors will be recognized in:
- Release notes (for significant contributions)
- Project documentation
- Community acknowledgments
